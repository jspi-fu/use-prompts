
## RIPER-5 使用方法

### 核心概念

RIPER-5 是一个五阶段工作流协议，通过强制 AI 按照固定模式工作，确保代码修改的安全性和可控性。

### 五个模式

1. **RESEARCH（研究模式）**
   - **目的**：信息收集和深入理解
   - **允许**：阅读文件、提出澄清问题、理解代码结构、分析系统架构
   - **禁止**：建议、实施、规划、任何行动或解决方案的暗示
   - **默认模式**：每次新对话开始时默认处于此模式

2. **INNOVATE（创新模式）**
   - **目的**：头脑风暴潜在方法
   - **允许**：讨论多种解决方案想法、评估优势/劣势、寻求方法反馈、探索架构替代方案
   - **禁止**：具体规划、实施细节、任何代码编写、承诺特定解决方案

3. **PLAN（规划模式）**
   - **目的**：创建详尽的技术规范
   - **允许**：带有精确文件路径的详细计划、精确的函数名称和签名、具体的更改规范、完整的架构概述
   - **禁止**：任何实施或代码编写、甚至可能被实施的"示例代码"、跳过或缩略规范
   - **必需输出**：编号的、顺序的实施清单

4. **EXECUTE（执行模式）**
   - **目的**：准确实施模式3中规划的内容
   - **允许**：只实施已批准计划中明确详述的内容、完全按照编号清单进行、标记已完成的清单项目
   - **禁止**：任何偏离计划的行为、计划中未指定的改进、创造性添加或"更好的想法"
   - **进入要求**：只有在明确的"ENTER EXECUTE MODE"命令后才能进入

5. **REVIEW（审查模式）**
   - **目的**：无情地验证实施与计划的符合程度
   - **允许**：逐行比较计划和实施、已实施代码的技术验证、检查错误、缺陷或意外行为
   - **必需**：明确标记任何偏差、验证所有清单项目是否正确完成、检查安全影响

### 使用流程

1. **初始化**
   - AI 默认处于 RESEARCH 模式
   - 创建功能分支（如需要）：`git checkout -b task/[TASK_IDENTIFIER]_[TASK_DATE_AND_NUMBER]`
   - 创建任务文件（如需要）：`mkdir -p .tasks && touch ".tasks/${TASK_FILE_NAME}_[TASK_IDENTIFIER].md"`

2. **模式转换**
   - 必须使用明确的模式转换信号：
     - "ENTER RESEARCH MODE"
     - "ENTER INNOVATE MODE"
     - "ENTER PLAN MODE"
     - "ENTER EXECUTE MODE"
     - "ENTER REVIEW MODE"
   - 没有这些确切信号，AI 应保持在当前模式

3. **执行工作流**
   ```
   RESEARCH → INNOVATE → PLAN → EXECUTE → REVIEW
   ```
   - 每个阶段必须完成其目标后才能进入下一阶段
   - EXECUTE 模式发现需要偏离计划时，自动回到 PLAN 模式
   - 所有实施完成且用户确认成功后，从 EXECUTE 转到 REVIEW

4. **任务文件管理**
   - 任务文件记录整个执行过程
   - 包含：背景、任务描述、分析、提议的解决方案、任务进度、最终审查
   - 位置：`.tasks/${TASK_FILE_NAME}_[TASK_IDENTIFIER].md`

### 关键协议规则

- **模式声明**：AI 必须在每个响应的开头用方括号声明当前模式，格式：`[MODE: MODE_NAME]`
- **严格遵循**：在 EXECUTE 模式中，必须 100% 忠实地遵循计划
- **偏差标记**：在 REVIEW 模式中，必须标记即使是最小的偏差
- **禁止自主决策**：在声明的模式之外，AI 没有独立决策的权限
- **语言设置**：除非用户另有指示，所有常规交互响应都应该使用中文

### 适用建议

**选择 `global.md` 如果**：
- 你的 AI 助手表现稳定，很少出现过度修改
- 你希望更快的开发节奏
- 你能够及时审查和修正 AI 的输出
- 项目对代码变更的容错性较高

**选择 `RIPER5.md` 如果**：
- AI 助手经常未经授权修改代码
- 项目对代码稳定性要求极高
- 你需要严格控制每个变更步骤
- 你希望有完整的任务记录和审计轨迹
